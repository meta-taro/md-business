### B. 列の `type` 列挙

初期は **7 種**:

| type | Sheets での実装 | Markdown 側 |
|---|---|---|
| `text` | DataValidation なし | 単純セル |
| `multiline_text` | DataValidation なし、改行可 | export 時に `<br>` → `\n` 変換 |
| `enum` | `requireValueInList(values).setAllowInvalid(false)` | values 配列を frontmatter 宣言 |
| `date` | `requireDate()` | ISO-8601 (`YYYY-MM-DD`) |
| `number` | `requireNumberBetween(min, max)`（min/max 任意） | 数値リテラル |
| `checkbox` | `insertCheckboxes()` | `[x]` / `[ ]` |
| `url` | DataValidation なし（v0.7.1 で URL 検証検討） | 単純テキスト |

**合意したい点**: 上記 7 種で v0.7.0 の十分性を確認。他に `boolean` / `time` / `currency` / `percentage` を初版から入れるべきか、後発でも問題ないか。

### D-1 確定設計（イレギュラー通知）

Markdown 優先で捨てる/Sheets 取り込みの二択ではなく、**Markdown スキーマと Sheets 実データの差分を検出して通知する**。

| 検出対象 | 通知方法 | 復旧 |
|---|---|---|
| シート側に未知列が追加された | サイドバーに警告 + 該当列ヘッダにメモ（`SpreadsheetApp.Range.setNote`） | PdM が「Markdown に列追加」or「シート列を削除」を選択 |
| `enum` 値が `values` 外（例: `OK/NG/保留/未実施` の枠を超えた `?`） | 当該セルに薄黄色 + 警告メモ + サイドバーに件数表示 | PdM がセル修正 or `values` を frontmatter で拡張 |
| `date` 型に非日付（手動入力ミス） | 当該セルに薄赤 + 警告メモ + サイドバーに件数表示 | PdM がセル修正 |
| `number` 型に文字列 | 当該セルに薄赤 + 警告メモ + サイドバーに件数表示 | PdM がセル修正 |
| `checkbox` 型に boolean 以外 | （DataValidation で防ぐので発生しないはず、防御として検出） | – |
| 必須列の空セル（将来 `required: true` 対応時） | 当該セルに薄赤 + サイドバーに件数表示 | PdM が入力 |

実装方針:

- `apps/google-workspace-addon/src/lib/validateSheet.ts`（新規）
  - 入力: Sheets `getValues()` + frontmatter `columns`
  - 出力: `ValidationResult[] = { row, col, columnName, type: 'unknown_column' | 'enum_out_of_values' | 'invalid_date' | ..., currentValue, expected, message }[]`
- import 時（md → Sheets）: validation 実行不要（Markdown が真なので常に整合）
- export 時（Sheets → md）: validation 実行 → 結果をサイドバーに表示 → `markdownBody` に追加するか否かは PdM 操作（OK ボタンで export 強行 / キャンセル）
- onEdit 自動同期時: validation 実行 → 結果をシートに **メモ（`setNote`）** + サイドバーに「⚠ N 件のイレギュラー」表示 → GitHub commit はバリデーション通過行のみ反映（イレギュラー行は手元に通知のみで commit 保留）

#### GitHub 連携の認証

- PdM が GitHub Personal Access Token（または Fine-grained PAT）を発行（`contents: write` スコープ、対象リポ限定推奨）
- Apps Script の `PropertiesService.getUserProperties().setProperty('GITHUB_PAT', '...')` に保存（**baseline §15: PdM 手作業**）
- 万一 PAT が漏洩しても影響範囲は `contents: write` の対象リポのみ。GitHub 側で revoke 可能

#### onEdit installable trigger

- Apps Script Editor の「トリガー」UI で `onSheetEdit` を installable trigger として登録（**PdM 手作業**、baseline §15）
- ScriptApp.newTrigger プログラム的登録もコード経由で可能だが、初回は OAuth 再同意モーダルが走るため PdM がブラウザで実行する必要あり
- 編集 → debounce 2 秒 → `validateSheet` → 通過行のみ `sheetToSchema` で md 再生成 → `UrlFetchApp.fetch(githubApi, ...)` で `PUT /repos/{owner}/{repo}/contents/{path}` 実行

- **v0.7.0**: 最終編集勝ち（`If-Match` ヘッダなし、`sha` チェックなしで上書き）。ローカル md を編集して同時に Sheets も編集すると、Sheets 側が GitHub commit を上書きする可能性あり。PdM 1 人運用なら問題なし
