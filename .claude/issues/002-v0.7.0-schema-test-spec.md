```yaml
---
schema: test-spec/v1
documentNumber: TEST-2026-001
title: ログイン機能 検証シート
version: 0.1.0
issueDate: "2026-06-18"
status: executing       # draft | review | executing | completed

  - { name: 田中 正智, role: 検証担当 }

  - ./../spec/login-system.md       # 基本設計書（schema-spec v0.5.0）への参照
googleSheetId: "1abc...xyz"         # 連携先 Sheets ID（双方向同期で使用、未設定時は import/export のみ）
columns:
  - { name: ID,         type: text }
  - { name: 検証項目,    type: text }
  - name: 期待結果

  - name: 備考

---

| ID    | 検証項目                          | 期待結果              | 結果   | 担当 | 実施日     | 備考       |
|-------|-----------------------------------|----------------------|--------|------|------------|------------|
| T-001 | 正しいパスワードでログイン成功      | ホーム画面へ遷移      | OK     | 田中 | 2026-06-18 |            |
| T-002 | 誤ったパスワードでエラー表示        | エラーメッセージ表示  | NG     | 田中 | 2026-06-18 | 想定外成功 |
| T-003 | 存在しない ID でエラー表示          | ユーザ未登録メッセージ | 未実施 |      |            |            |
```

**合意したい点**:

- (A-1) `schema: test-spec/v1` ヘッダで良いか（既存パッケージは `schemaVersion: spec/v1` だが PdM 議論 2026-06-16 では `schema: ...` 形式が AI 親和性高いとの示唆）
- (A-2) `columns` を必須にして「列スキーマ未宣言」を warning にする、または columns 省略時は本文 table のヘッダ行から自動推測（type=text 統一）する fallback を持つか
- (A-3) `googleSheetId` の役割: 未設定時は手動 import/export のみ、設定時は onEdit 自動同期（v0.7.0 では未実装、v0.7.1 以降）の予約フィールドとしておく方針で良いか

### B. 列の `type` 列挙

初期は **7 種**:

| type | Sheets での実装 | Markdown 側 |
|---|---|---|
| `text` | DataValidation なし | 単純セル |
| `multiline_text` | DataValidation なし、改行可 | export 時に `<br>` → `\n` 変換 |
| `enum` | `requireValueInList(values).setAllowInvalid(false)` | values 配列を frontmatter 宣言 |
| `date` | `requireDate()` | ISO-8601 (`YYYY-MM-DD`) |
| `number` | `requireNumberBetween(min, max)`（min/max 任意） | 数値リテラル |
| `checkbox` | `insertCheckboxes()` | `[x]` / `[ ]` |
| `url` | DataValidation なし（v0.7.1 で URL 検証検討） | 単純テキスト |

**合意したい点**: 上記 7 種で v0.7.0 の十分性を確認。他に `boolean` / `time` / `currency` / `percentage` を初版から入れるべきか、後発でも問題ないか。

### C. 視覚スタイル（`visual:`）の記法

```yaml
- name: 結果
  type: enum
  values: [OK, NG, 保留, 未実施]
  visual:
    NG:   { row_background: "#FFE5E5" }   # 行全体の背景色
    OK:   { background: "#E5F5E5" }       # セル単独の背景色
    保留: { background: "#FFF5D5", color: "#996600" }  # 文字色も指定可
```

**合意したい点**:

- (C-1) `row_background`（行全体）と `background`（セル単独）の 2 キーで区別する設計で良いか
- (C-2) Apps Script の `setConditionalFormatRules` 実装: 行全体 = `whenFormulaSatisfied('=$結果列="NG"')` + 全列レンジ / セル単独 = `whenTextEqualTo('OK')` + 該当列のみ。これで正しい
- (C-3) Markdown export 時に `visual:` を完全保持して書き戻す（= ラウンドトリップ可能）。シート側で書式が手動編集された場合は frontmatter が真として再 import 時に上書き

### D. 双方向同期の責務分離（最重要）

| 何が真か | 編集権 | export → import で復元されるか |
|---|---|---|
| **列スキーマ（`columns`）** | Markdown frontmatter が真 | ✅ |
| **視覚スタイル（`visual`）** | Markdown frontmatter が真 | ✅ |
| **データ検証ルール（DataValidation）** | Markdown frontmatter が真 | ✅ |
| **行データ（テスト項目 / 結果 / 担当 / 実施日 / 備考）** | **Sheets が真** | ✅ |
| **シート側で追加された未知の列** | ❓ | ❓ → **PdM 合意必要** |

**合意したい点（D-1）**: シート側で勝手に追加された列の扱い

- 案 X: **Markdown 優先** — export 時に未知の列は捨てる（シート側で実験的に追加したものは保持されない）。実装シンプル、frontmatter が常に真
- 案 Y: **Sheets 取り込み** — export 時に未知の列を `columns` に `type: text` で追記して frontmatter に書き戻す。柔軟だが「frontmatter が真」の原則と矛盾
- 推奨: **案 X**（schema-test-spec のコンセプトは「列スキーマは Markdown で設計」なので、シート側はあくまで「データ入力面」）
