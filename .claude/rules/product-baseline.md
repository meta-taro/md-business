# プロダクト共通ベースルール（正本）

> このファイルは全プロダクト共通の「守ってもらうルール」の **正本** です。
> 各プロダクトリポジトリへ配布される際に `.claude/rules/product-baseline.md` として焼き込まれます。
>
> 基本方針: **「速く作るより、壊さない仕組みを最初に作る」**。

---

## 1. パッケージマネージャ（ハード禁止あり）

- **pnpm のみ使用**。
- **禁止**: `npm install` の混在 / `package-lock.json` 生成 / `yarn.lock` 生成。
- 理由: 速度・ディスク効率・monorepo 対応・モダンフロントエンド標準、およびサプライチェーン防御。

## 2. 技術選定の均衡

- 「古すぎる安定版」ではなく「モダンで実務的」な構成を優先。
- ただし実験的すぎる技術は避ける。バージョン指定は必ず確認する。

## 3. ローカル開発環境が前提

- 整備必須: `.env.example` / README のセットアップ手順 / pnpm scripts / Git hooks / テストコマンド / ローカル起動手順。

## 4. TDD（テスト駆動開発）を前提

- 目的は「過去の機能破壊を防ぐ」こと。
- **Next.js / TypeScript 推奨スタック**: Vitest / React Testing Library / ESLint / TypeScript typecheck / Playwright / Storybook
  - 実行例: `pnpm lint && pnpm typecheck && pnpm test:run`
- **Go 推奨スタック**: gofmt / go vet / go test / golangci-lint / staticcheck / templ generate（使用時）

## 5. 品質ゲート（Git hooks）

- **pre-commit（壊れたコードを commit させない）**: format / lint / typecheck / unit test
- **pre-push（壊れたブランチを push させない）**: build / storybook:build / test:run

## 6. commit と push の分担

- **commit = AI エージェント**（フェーズごとに小さく）。
- **push = 人間**（確認・責任）。**人間確認なしの push 禁止**。
- push 前確認項目:
  - 実際の動作確認 / 余計なファイル変更がないか / README と実装のズレ / テスト成功 / セキュリティ問題 / commit の粒度
  - 変更に対応するテストが追加されているか。不要な場合は理由を説明する。

## 7. 作業の進め方

- 実装前に **`/plan` で計画**を立てる。確認: 技術が古すぎないか / フェーズが大きすぎないか / テストが後回しでないか / ドキュメント更新が含まれるか / CI・Git hook が後回しでないか。
- **ローカル Issue で管理**: `.claude/issues/001-xxx.md` 形式。各 Issue に「目的・作業内容・完了条件・確認コマンド・作業ログ・結果」を記載。
- 小さいフェーズで作業。
- **禁止**: テスト削除 / 無関係ファイルの変更 / テストの後回し。

## 8. コード品質（人間が読めるコード）

- 関数・クラスの責務を小さく。意味の分かる変数・関数名。
- 複雑な条件分岐・ビジネスルール・非自明な決定にはコメント（なぜ必要かを書く）。
- 一時的実装は TODO で明記。巨大な関数・ファイル・モノリシック実装を避ける。

## 9. アーキテクチャ（DDD / Clean Architecture）

- `domain`: 業務ルール・エンティティ・値オブジェクト
- `usecase`: アプリケーション処理手順
- `infrastructure`: DB・外部 API・ファイル・メール
- `presentation`: Controller・API Route・UI
- `tests`: 単体・結合テスト
- Controller などに直接業務ロジックを入れない。**小規模プロジェクトで過剰設計しない**。

## 10. ドキュメント同期

- 進捗管理ファイルを最新に保つ: `CLAUDE.md` / `.claude/project-status.md` / `.claude/roadmap.md` / `.claude/decisions.md`
  - 記録項目: 現在フェーズ・完了/未完了作業・次タスク・技術的決定・注意点・テスト状況・既知問題。
- セットアップ手順・フェーズ完了・技術的決定の変更時は該当ドキュメントを更新。
- **テスト未実装では「完了」と表記しない**。

## 11. UI を含む場合は DESIGN.md

- カラーパレット / フォントサイズ / 余白 / 角丸 / ボタン・カードの見た目 / レイアウト方針 / レスポンシブ方針 / UI 雰囲気を明文化。

## 12. 依存・セキュリティ確認

- 新規パッケージ導入前に確認: 依存の安全性 / 不要な外部送信の有無 / APIキー・秘密情報の扱い / GitHub Actions の危険処理 / ライセンス。

## 13. 確認プロンプトの許可範囲

- プロジェクト進行に応じて確認プロンプトを精査し、「毎回 Yes / 都度確認 / 人間確認必須 / No」に分類して効率化する。許可申請時は対象コマンド・判断理由・影響範囲を整理する。

## 14. PdM ⇄ エンジニア間の通信（GitHub Issues 双方向）

- **PdM・司令塔エージェント ⇄ 担当エンジニア（AI 含む）間の連絡は、当該リポの GitHub Issues で双方向に行う**。Slack / Discord / メール / チャットアプリは使わない。
- 連絡があった日のみ `[YYYY-MM-DD] {内容}` 形式で新規 issue を立てる。追加発話は同 issue へコメント。**1日1 issue のノルマではない**。話がない日は何も書かない（沈黙＝OK）。
- AI エージェントは `gh issue create` / `gh issue comment` で機械的に起票・返信してよい。**人間が文面整形する必要はない**。
- 司令塔から起票する issue は **機械的に判定できる完了条件があるタスクに絞る**（質の悪い形式記録をノルマで書かせない）。
- 進捗の記録は引き続きリポ内ファイル（`CLAUDE.md` / `.claude/project-status.md` / `.claude/roadmap.md` / `.claude/decisions.md` / `LEARNING.md`）で行う（項目10「ドキュメント同期」のとおり）。司令塔エージェントは `git pull` して直接読むため、これらの内容を issue にコピペする必要はない。
- 完了報告も「コード/ドキュメントを更新した上で、必要なら GitHub Issues に短く一言コメント」で十分（長文の進捗レポートは不要）。
- 定例・日次メール・週次報告のような **時間ベースの報告ノルマは設けない**。司令塔エージェントが必要に応じてリポを観察し、質問があれば issue で投げる。

## 15. Secrets / credential 操作は人間のみ（AI 自走禁止）

GitHub Repository / Environment / Organization secrets、および各種クラウド（GCP / AWS / Cloudflare / Neon / Supabase / Turso / Vercel 等）の credential 投入・更新・削除は、**必ず人間が実行**する。AI（Claude Code 含む）は以下を厳守する。

- ❌ `gh secret set` / `gh secret delete` / `gh secret list` 系の自走実行禁止
- ❌ AWS CLI / gcloud CLI / wrangler / supabase CLI / neonctl 等での credential 操作の自走実行禁止
- ❌ 「CI を緑にするため」「動作確認のため」等の理由で dummy 値を secrets に投入することも禁止
- ✅ 必要な secret を提案するのは OK（投入手順を提示するまで）
- ✅ 既に投入済みの secret を発見した場合は **報告のみ**、削除・変更しない

**理由**: 過去に AI が CI セットアップで dummy 値を Repository secrets に投入し、所有・監査が不能になる事例が発生したため。secrets は人間（PdM）が一元管理する。

## 16. アーキテクチャ決定の独断変更禁止

`PRD.md` / `.claude/decisions.md` / `.claude/roadmap.md` に記載された技術選定（DB / クラウド / フレームワーク / 言語等）を、AI が独断で変更してはならない。

- ❌ 決定済みの DB を別 DB に差し替える（例: Aurora DSQL → Neon、Turso → Supabase）
- ❌ 決定済みのクラウドを別クラウドに差し替える（例: AWS → Vercel、Cloudflare → GCP）
- ❌ 決定済みのスタックを別言語・別フレームワークに差し替える（例: Kotlin → Node.js）
- ❌ 「セットアップが楽だから」「無料枠が広いから」を理由に勝手に乗り換えるのも禁止
- ✅ 「決定された技術が要件に合わない」と気づいたら、**Issue で司令塔（PdM）に相談**する
- ✅ 司令塔の判断が出るまで作業を止める

**理由**: 技術選定は案件全体の戦略（例: 「この案件は AWS 初投入のリハーサル」「この案件は無料枠固執」）に紐づいており、AI が局所最適で変更すると上位戦略が崩れる。

## 17. 非エンジニア担当への作業依頼禁止

担当者の役割が `.claude/issues/` 内で **owner-task のみ実行可能**（= 非エンジニア）と明記されている場合、AI は以下を依頼してはならない。

- ❌ 外部サービスのアカウント作成（GCP / AWS / Neon / Supabase / Turso / Cloudflare 等）
- ❌ API キー・credential の発行
- ❌ Repository secrets / Environment secrets の投入
- ❌ コマンドラインでの credential 操作（`gh auth login` 等）
- ❌ `.env` ファイルの編集・credential 貼り付け
- ❌ パッケージインストール・ビルド・デプロイ

これらが必要な場合は、**司令塔（PdM）に Issue で依頼**する。担当者には「PRD / 方針の意思決定」「Issue へのコメント」「画面・出力の確認」のみを依頼する。

**理由**: 非エンジニア担当者は技術判断ができないため、AI の指示に従って外部アカウントを作成すると、個人アカウントに業務インフラが侵食する事故が発生する（過去に発生済み）。

## 18. AI エージェントの Issue 反応義務（沈黙禁止）と セッション開始/終了の判定

PdM・司令塔から GitHub Issues 経由で連絡が来た場合、AI エージェントは **読んだことが PdM 側から確認できる形でコメント反応** する。沈黙は「読んでいない」「ストール中」「無視」と区別がつかず、マネジメント不能になるため厳禁。

### セッション開始の判定（明示 / 暗黙）

以下のいずれかで「セッション開始」とみなし、開始時必須動作を実行する:

- **明示トリガー**: ユーザーから「おはようございます」「本日の作業を開始します」「作業始めます」「再開します」等の開始挨拶を受けた
- **暗黙トリガー**: 明示挨拶がなくても、前回入力から **日付が変わって** おり、現在時刻が **9:00〜22:00** の場合、内容を問わず開始扱い
- **対象外**: 22:00〜翌 9:00 の初入力は緊急扱い・開始時必須動作はスキップ可

### 開始時必須動作（順序固定）

1. `git pull --ff-only` で `.claude/` 配下と CLAUDE.md / baseline を最新化
2. `gh issue list --state open` で open issue 一覧確認
3. `gh issue list --state closed --limit 10` で直近の close も確認（reopen / 続編 issue を見落とさないため）
4. 自分宛（assignee 含む）または 自リポの open issue は **全件 `gh issue view <番号>` で本文確認**
5. 応答冒頭で挨拶 + 確認結果を返す:
   - **ユーザーが開始挨拶していた場合**: 「Open Issue を X 件確認しました（#A, #B）。本日は #N から着手します。」
   - **ユーザーが挨拶せず作業内容を投げた場合（暗黙トリガー発火）**: 「**おはようございます。本日初の作業として処理します。** Open Issue を X 件確認しました（#A, #B）。先に #N から着手します。」
   - **しつこさ防止**: ユーザーが既に挨拶していたら、AI 側から挨拶を重ねない

### 日中の Issue 確認（任意・必須ではない）

開始時に確認済なら、日中の追加確認は必須ではない。長時間（数時間）作業継続中に新規 Issue が起票されている可能性が高ければ、`gh issue list --state open` で 1 回だけ差分を確認してよい。**毎ターン確認は禁止**（過剰）。

### 起票・指示変更への反応（着手前）

- 新規起票された Issue は、作業開始前に最低1回はコメント反応:
  - 「読みました。着手します（目安: 〇〇まで）」
  - 「読みました。質問: 〇〇」
  - 「読みました。優先度的に 〇〇 の後になります」
- **沈黙のまま着手 / 沈黙のまま無視は禁止**
- 追記コメント（指示変更・差し戻し）は、気づき次第「読みました」を明示してから作業継続
- 自分が close した Issue が PdM 側で reopen されたら、必ず再読してから対応

### git pull のタイミング（3 箇所）

baseline / CLAUDE.md の正本は司令塔リポにあり、配布リポへは origin/main 経由で配布される。AI は以下のタイミングで pull する:

1. **セッション開始時**: `git pull --ff-only`（必須・開始時必須動作 手順1）
2. **AI が commit する直前**: `git pull --rebase --ff-only`（必須・origin の baseline 更新を取り込む。push 前のコンフリクト予防も兼ねる）
3. **人間 push 時**: ff エラーになったら pull --rebase してから再 push

これにより、司令塔が日中に baseline を更新しても、AI の次の commit 時点で自動的に取り込まれる。担当者（非エンジニア owner-task 担当）は何もしなくてよい。pull でコンフリクトが起きたら作業を止めて Issue で司令塔（PdM）に相談する。

### セッション終了の判定

- **明示トリガー**: ユーザーから「進捗を記録してください」「お疲れさまでした」「終わります」「今日はここまで」等の終了挨拶を受けた
- **AI からの終了確認**（相手未実施時のみ・しつこさ禁止）: 1 タスク完了 + 次指示がない明らかな区切りなら、AI 側から **同セッション内で 1 回だけ** 「ここで一旦区切りますか？進捗を記録しますか？」と確認してよい。**毎タスク毎に聞くのは禁止**

### 終了時必須動作

1. `.claude/project-status.md` を最新化
2. 該当 Issue にコメント or `gh issue close`（完了なら **AI 自身が `gh issue close` でクローズ**、PdM が手動 close する運用は禁止）
3. close 時にコメントで「完了内容・確認方法・関連コミット/PR」を残す
4. 一部完了の場合は close せず、進捗コメントを残して open のまま継続
5. `.claude/decisions.md` に当日の技術的決定があれば追記
6. 最終応答テンプレ:
   > 進捗記録完了。本日 #X, #Y を進めました（#X close、#Y は ... の状態で継続）。お疲れさまでした。

### Close を見落とさない

- PdM 側で close された Issue（自分が assignee or 自リポ）は、**close コメントを必ず読んで意図を理解**する
- 「PdM が close した = 続行不要」とは限らない（別 Issue に分割された / 方針変更で破棄された 等）。close コメントを読み、不明点があれば **新規 Issue または同 Issue にコメントで確認**
- 過去に AI が close を見落としたまま作業継続し、無効な実装を進めた事例があるため、**close コメントの読了は厳守**

**理由**: AI エージェントの作業状態は外部から不可視のため、PdM 側で進捗・指示理解を確認できる手段が Issue コメントしかない。沈黙はマネジメント不能を意味する。また、担当者が起動・終了の操作を意識せずとも、開始/終了の挨拶フレーズと commit 時 pull だけで baseline / Issue が確実に同期される仕組みにすることで、非エンジニア担当でも運用可能にする。
